对比其他方案

AI 浏览器方案越来越多。这一页诚实对比 chrome-use 和最有代表性的两个—— ego-lite（自建 Chromium 浏览器）和 browser-use（自动化框架）——并公开我们持续「试用 → 对比 → 把好的搬过来」的 workflow。 对比是活的：结论会随每一轮实测更新。

一句话定位

chrome-use 驱动你现有的真 Chrome——你的 profile、你的扩展、你的登录，什么都不用装。 ego-lite 是一个全新的浏览器：架构上很干净（agent 有自己的隔离 Space），但要你安装并信任一个新 App。 这就是核心取舍——零切换成本 vs 结构性隔离。我们的立场：把「真 Chrome 里的隔离」做到极致，而不是去 fork 一个浏览器。

我们在学它什么（并搬过来）

ego-lite 有几个想法确实值得学。我们逐项试用、对比、搬进 chrome-use 的方式（前提：始终驱动你的真 Chrome，绝不 fork 浏览器）。 进度和每一轮结论都记录在追踪 issue 里： #89 Learn from ego-lite。

  • 一等公民隔离上下文（对标 Space）——把 session 升成带独立标签的隔离上下文。
  • 所有权 + 交接状态机——显式 owner: agent | user | handed-off + handoff/resume，登录/2FA/验证码时干净地把控制权交还用户。（最高价值，优先做）
  • 登录继承——继承当前 profile 登录态的隔离模式，常见场景无需搬 cookie。
  • 富文本编辑器 write-probe 纪律——已搬，见 Canvas / WebGL 页末。
  • 快照内联 url= / 按名复用会话 / token 经济——已基本具备（--urls、--session-name）。

对比 workflow

这一页是活文档。每学一个竞品能力，我们走同一套流程，保证对比诚实、可复现：

  1. 试用——同一个真实任务，两边各跑一遍，记录命令、结果、耗时、token。
  2. 对比——判断是「我们已有」「基本持平」还是「真缺口」；持平就如实标注，不硬搬。
  3. 搬——真缺口就用 chrome-use 原生的方式实现（守住零安装护城河），落地后回填 issue #89 并更新本页表格。
✅ 原则
只搬能在你的真 Chrome 里实现的优点。任何要求「装一个新浏览器」的路径都不搬——那正是我们的护城河。
