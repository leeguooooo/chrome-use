对比其他方案

AI 浏览器方案越来越多。这一页诚实对比 chrome-use 和最有代表性的两个—— ego-lite（自建 Chromium 浏览器）和 browser-use（自动化框架）——并公开我们持续「试用 → 对比 → 把好的搬过来」的 workflow。 对比是活的：结论会随每一轮实测更新。

一句话定位

chrome-use 驱动你现有的真 Chrome——你的 profile、你的扩展、你的登录，什么都不用装。 ego-lite 是一个全新的浏览器：架构上很干净（agent 有自己的隔离 Space），但要你安装并信任一个新 App。 这就是核心取舍——零切换成本 vs 结构性隔离。我们的立场：把「真 Chrome 里的隔离」做到极致，而不是去 fork 一个浏览器。

我们在学它什么（并搬过来）

ego-lite 有几个想法确实值得学。我们逐项试用、对比、搬进 chrome-use 的方式（前提：始终驱动你的真 Chrome，绝不 fork 浏览器）。 进度和每一轮结论都记录在追踪 issue 里： #89 Learn from ego-lite。

  • 所有权 + 交接状态机——已实现：session handoff / resume / status / list；交接给用户后，agent 对该会话的一切驱动命令被拒绝（登录/2FA/验证码时干净地把控制权交还用户）。
  • 一等公民隔离上下文（对标 Space）——现有每会话彩色标签组 + 多 agent 隔离已提供隔离；新增 session list 把隔离会话（含所有权）显式列出。
  • 登录继承——架构固有：relay 模式驱动你的真 Chrome，agent 标签就在你的 profile 里，天然带你的登录；跨账号才用 cookie-use。
  • 富文本编辑器 write-probe 纪律——已搬，见 Canvas / WebGL 页末。
  • 快照内联 url= / 按名复用会话 / token 经济——已基本具备（--urls、--session-name）。

基准实测：性能与 token（核对"快 3 倍、更省 token"的说法）

ego 宣传"更快、更省 token"，坊间传成"快 3 倍"。我们实测同样的页面、同样的操作。结论：一个半真、一个恰好相反。

chrome-use 默认的 snapshot -i 只列交互元素，比 ego 默认的全结构树 省 1.8–4.5×。token 是 agent 真正花钱的地方——是我们更省，不是它。

6 个微操作 · ego 常驻运行时—≈0.00s

原始浏览器操作（导航、CDP）两边一样快。ego 的优势是常驻运行时：一个进程连续跑多个操作，没有每条命令的进程启动开销。 chrome-use 默认「一条命令一个进程」，每条约 0.15–0.17s 启动开销——所以微操作特别多的任务里 ego 会明显更快（"快 3 倍"多半来自这种形状）。 但典型任务（open + 快照 + 抽取）实测两边都 ~3s；而且 chrome-use 用 batch（一次往返、常驻守护进程）6 个操作 0.19s，基本追平 ego——还有 WebSocket 流做实时驱动。

ℹ️ 一句话
"快 3 倍"：只在「大量微操作 + 你天真地一条命令一个进程」时成立；典型任务持平，用 batch/流即可追平。"更省 token"：反了，默认工作流下我们省 1.8–4.5×。

对比 workflow

这一页是活文档。每学一个竞品能力，我们走同一套流程，保证对比诚实、可复现：

  1. 试用——同一个真实任务，两边各跑一遍，记录命令、结果、耗时、token。
  2. 对比——判断是「我们已有」「基本持平」还是「真缺口」；持平就如实标注，不硬搬。
  3. 搬——真缺口就用 chrome-use 原生的方式实现（守住零安装护城河），落地后回填 issue #89 并更新本页表格。
✅ 原则
只搬能在你的真 Chrome 里实现的优点。任何要求「装一个新浏览器」的路径都不搬——那正是我们的护城河。
