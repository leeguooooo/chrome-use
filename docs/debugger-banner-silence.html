静音 "chrome-use" started debugging this browser 提示条

类型 调研 + 方案（已落地） 状态 已解决：扩展 0.5.5+ 只 attach agent 自建标签页（用户标签页不再触发）；方案 A 已落地为 extension connect --silent 关联 ab-connect relay 路径 · 它怎么工作 · 故障排查 Q&A 日期 2026-06-24（调研）· 2026-07-06（更新为已解决）

用户想在 relay（扩展 connect）模式下驱动自己已登录的真实 Chrome，但每次都被 Chrome 顶部那条 "chrome-use" started debugging this browser [Cancel] [×] 提示条打扰，且右上角的 × 关不掉。本文说明它的本质、为什么关不掉，以及三个落地方案的取舍。

现在怎么做

大多数人根本不用管这条提示条。 扩展 0.5.5 起，agent 只 attach 自己新建的标签页， 你正在看的标签页不会被 attach，也就不会弹这条横幅（链路原理见 它怎么工作）。

万一你确实撞见它（某个 agent 标签页被 attach、横幅碍事），跑一次下面这条，在提示处按回车即可：

chrome-use extension connect
# 提示：Restart Chrome now? [Y/n]  ← 直接回车 = 重启
# Chrome 一次性重启（约 2 秒），标签和登录自动恢复，这台 Chrome 整段会话不再有横幅
  • chrome-use extension connect --silent：跳过确认，直接重启。
  • chrome-use extension connect --keep-banner：完全不碰浏览器，保留横幅。
  • 默认（不带 flag）在交互终端里会先问一句 Restart Chrome now? [Y/n]，回车即同意；非交互环境不动浏览器，只打印一行提示。

v1.5.72 起：装机时问一次，之后 agent 自动处理。 curl … | sh 装完二进制会跑一遍引导式 extension install（借 /dev/tty，管道里也能问真人），并问一句「允许 chrome-use 需要时重启 Chrome 去掉横幅吗？」。 答应后写入 ~/.chrome-use/silence-consent；此后非交互的 agent 连接会凭这份同意自动静默、不再问—— 解决了「agent 没 TTY、问不了人，横幅只能一直留着」。安全护栏不变：只在单实例时动、优雅退出、--restore-last-session 恢复标签。 想跳过装机引导：AGENT_BROWSER_NO_SETUP=1。

实现见 cli/src/silence.rs。

以下是当时（2026-06-24）的原始调研，记录这条提示条的本质、为什么关不掉， 以及在落地成上面这条命令之前对三个方案的取舍推演。结论已落地，下文只为解释「为什么是这样」。

1 · 这条提示条是什么

它不是常见的「Chrome 正受自动化测试软件控制」黄条。那条由 enable-automation 开关触发；这条来自 chrome.debugger 扩展 API —— 任何扩展调用 chrome.debugger.attach() 时，Chrome 都会强制弹出 <扩展名> started debugging this browser，以此告知用户「有扩展正在以调试器权限读写本浏览器」。

在本项目里，这条提示由 relay 链路触发：

位置行为
extensions/ab-connect/background.js:401 chrome.debugger.attach(dbg, '1.3') —— 提示条的直接来源
extensions/ab-connect/background.js:268 / 412 之后所有 CDP 命令走 chrome.debugger.sendCommand(...)，提示条全程驻留

2 · 为什么 × 关不掉（这是 Chrome 的设计）

  • 这条 infobar 唯一有效的动作是 Cancel —— 点了会 detach 调试器， 也就是直接断开我们的自动化。
  • 右上角的 × 顶多视觉上收一下，只要调试会话还在，Chrome 会再弹回来。
  • 不存在「保留自动化 + 隐藏提示」的运行时手段：JS、CDP、扩展 API 都改不了一个已经在运行的 Chrome 进程的这条提示。这是安全设计，刻意如此。

3 · 唯一干净的抑制方式：启动开关

Chrome 有一个命令行开关 --silent-debugger-extension-api，专门静音这条 chrome.debugger 提示条。关键约束：

它必须在 Chrome 启动那一刻就带上，无法注入到已经运行的 Chrome。 而 relay 路径恰恰是 attach 到「用户自己已经开着的」普通 Chrome —— 所以要静音，就得让那个 Chrome 带着这个 flag 冷启动一次。

对照：为什么 --profile / --launch 自启路径没有这条

项目里 chrome-use 自己拉起 Chrome 的路径走的是 --remote-debugging-port（CDP 直连，见 cli/src/native/cdp/chrome.rs:209），根本不用 chrome.debugger 扩展 API， 因此天然没有这条提示条。问题只发生在 relay 路径。

4 · 三个方案及取舍

方案做法体验代价
A 推荐 在 extension connect 加「静音模式」：检测到真实 Chrome 在跑就提示用户，确认后 Cmd-Q 全退，再带 --silent-debugger-extension-api 冷启动同一个 user-data-dir。 一次重启后彻底无提示条；profile 不变 → 所有登录态保留。最贴合「想用但不想被提示」。 需整体退出 + 冷启一次，打断用户当下窗口状态（标签页由 Chrome 自身恢复）。
B 永久设置：让用户固定用带 flag 的方式开 Chrome（LaunchAgent / shell alias / 改快捷方式）， 例如 open -a "Google Chrome" --args --silent-debugger-extension-api。 装一次，之后每个 session 都无提示。 macOS 上 --args 只在冷启动生效；从 Dock 正常点开就不带 flag 了。安装有门槛、易踩坑。
C 不动提示条，只把文案讲清楚（在 skill / connect 输出里说明「这条是 Chrome 安全提示，× 关不掉， 点 Cancel 会断开」）。 零改动、零风险。 没解决用户诉求，每次仍被打扰。

5 · 推荐：方案 A（为小白设计）

设计第一原则：用户什么 flag 都不碰。 不是教用户敲 Cmd-Q + open --args（那是开发者操作，小白做不到）， 而是 chrome-use 自己检测、自己重启、自己恢复。用户的全部操作 = 看到一句大白话，按一下回车。

用户视角：从 4 步降到 0 步

① Cmd-Q 彻底退出 Chrome
② 终端敲 open -a "Google Chrome" --args --silent-debugger-extension-api
③ 担心标签页丢了
④ 再手动 extension connect		 ① 照常用 chrome-use →
工具弹一句：「需要重启 Chrome 去掉提示，标签和登录都会自动恢复。现在重启？[回车=好]」
② 按回车。完。

交互形态（默认回车=同意）

# chrome-use 检测到：真实 Chrome 在跑 + 未带 flag + 会弹提示条 → 自动提示：

  为去掉 Chrome 顶部那条「调试提示」，需要把 Chrome 重启一次。
  你的标签页和登录都会自动恢复，大约 2 秒。现在重启？[回车=好 / n=不用]

# 回车 → 工具全自动：
#   1. 优雅退出 Chrome（AppleScript `quit`，让 Chrome 自己存会话）
#   2. 冷启动同一 profile，附带：
#        --silent-debugger-extension-api   ← 静音提示条
#        --restore-last-session            ← 强制恢复刚才的标签页
#   3. 重新连上 relay
# 结果：Chrome 闪一下，标签 + 登录原样回来，顶部提示条消失。

实现要点（已落地，见 cli/src/silence.rs）

  • 标签不能丢（小白最在意的）：重启时加 --restore-last-session，强制恢复上次会话， 不依赖用户有没有把「启动时」设成「继续浏览上次打开的网页」。这是把自动重启变安全的关键一招。
  • 定位 profile：复用 get_chrome_user_data_dirs()（cli/src/native/cdp/chrome.rs:901）， 重启时带回原 --user-data-dir 与原 --profile-directory，确保是同一份登录态。
  • 优雅退出而非 kill：用 AppleScript tell application "Google Chrome" to quit（等价 Cmd-Q）， 让 Chrome 落盘保存会话；轮询确认进程真的退出后再冷启。绝不 kill -9（会丢会话）。
  • 幂等 / 不打扰：若检测到 Chrome 已带 flag（进程 cmdline 里有它）→ 直接跳过，不重启、不提示。
  • opt-in + 记住选择：必须显式确认才重启，绝不静默退用户浏览器；同意后写入配置，后续同一 Chrome 会话不再追问 （flag 随进程存活，Chrome 不关就一直无提示，正常用户开着 Chrome 数天 → 实际只重启一次）。
  • 不要叠加 --remote-debugging-port：relay 路径靠扩展 + native messaging，不需要也不应开放 raw CDP 端口 （会触发另一个「Allow remote debugging?」弹窗，见 cli/src/native/cdp/chrome.rs:895-897）。本方案只加上面两个 flag。

诚实声明：--silent-debugger-extension-api 移除的是 Chrome 出于安全考量、 刻意展示的「有扩展在调试本浏览器」提示。对「用户在自己机器上、用自己的 agent 驱动自己的 Chrome」这一第一方自用场景是合理的； 但这一点应在文档与 --silent 的确认语里讲明，而不是悄悄抹掉。

6 · 跨平台（Windows / macOS / Linux）

用户不只在 Mac。两个 flag 本身是跨平台的 Chromium 开关，各平台都认 —— 需要分平台处理的只是「找 Chrome、优雅退出、冷启」这层 OS 管道。flag 拼接共用。

flags 全平台一致：--silent-debugger-extension-api --restore-last-session

各平台的坑（都关系到「会话别丢」）

  • Windows 最关键：绝不用 taskkill /F 或 PowerShell Stop-Process —— 那是强杀， Chrome 当成崩溃，--restore-last-session 会退化成弹「恢复」气泡甚至丢标签。必须用不带 /F的 taskkill，让 Chrome 正常落盘 last session，再配 --restore-last-session 才稳。
  • macOS：open --args 只在冷启生效，所以必须先确认完全退出再启。
  • 全平台共有：某页带 beforeunload（「确定离开？」）会卡住优雅退出 → 实现里要超时后提示用户， 而不是超时就强杀丢会话。

CLI 里 cli/src/native/cdp/chrome.rs 已有按 OS 解析 Chrome 路径的逻辑，退出/冷启层照同样的 #[cfg(target_os)] 分支写即可。

7 · 一句话结论

提示条来自 chrome.debugger，× 关不掉是 Chrome 故意的；运行时无解。唯一干净的静音是用 --silent-debugger-extension-api 冷启动用户的 Chrome。但别让用户自己敲命令 —— 让 chrome-use 自己检测、自己优雅重启、自己恢复标签（--restore-last-session）， 用户只需按一下回车：标签和登录原样回来，提示条消失。一次重启换这台 Chrome 整段会话无打扰。
